# Эффективное управление кодом: подробное руководство по Git Flow в Yalla

В современной разработке программного обеспечения Git стал неотъемлемым инструментом для контроля версий. Грамотно выстроенный рабочий процесс, или Git Flow, позволяет команде работать слаженно, избегать конфликтов и поддерживать высокое качество кода. В этой статье мы подробно разберем эффективный Git Flow, принятый в компании Yalla, основанный на двух основных и нескольких вспомогательных ветках, который идеально подходит для проектов с плановыми релизами.

## Основы рабочего процесса в Yalla: ветки master и develop

Центральными элементами Git Flow в Yalla являются две долгоживущие ветки: `master` и `develop`.

- **master (основная ветка)**: Эта ветка — источник истины. В ней содержится исключительно стабильный, протестированный и готовый к развертыванию на продуктивном сервере код. Прямые коммиты в `master` строго запрещены. Каждый коммит в этой ветке является, по сути, новой версией продукта и должен быть помечен соответствующим тегом (например, `v1.0.0`).
    
- **develop (ветка разработки)**: Эта ветка является основной для всей команды разработки. Она отражает последние изменения и содержит код, который готовится к следующему релизу. Именно из `develop` создаются ветки для новых функций и исправлений, и в нее же они потом вливаются. Ночной билд и автоматическое тестирование, как правило, настраиваются именно на эту ветку.
    

## Вспомогательные ветки: изоляция и порядок

Для организации параллельной работы, подготовки релизов и срочных исправлений используются вспомогательные ветки с четко определенным назначением и жизненным циклом.

### `feature/*` (ветки для новой функциональности)

- **Назначение**: Разработка новой функциональности. Каждая новая задача, будь то большая фича или небольшое улучшение, должна разрабатываться в отдельной ветке. Это обеспечивает изоляцию кода и предотвращает попадание нестабильных изменений в основную ветку разработки.
    
- **Создание**: Ветки `feature/*` всегда создаются из ветки `develop`. Именовать их следует осмысленно, например, `feature/user-authentication` или `feature/payment-gateway-integration`.
    
- **Кто создает**: Разработчики.
    
- **Жизненный цикл**: После завершения работы над новой функциональностью и прохождения код-ревью, ветка `feature/*` вливается обратно в `develop`. После слияния локальную и удаленную ветку `feature/*` следует удалить.
    

### `release/*` (ветки для подготовки релиза)

- **Назначение**: Подготовка новой версии продукта к выпуску. В этих ветках происходит стабилизация кода, исправление мелких ошибок, обновление документации и другие задачи, связанные с релизом. Создание `release` ветки "замораживает" функциональность, которая войдет в релиз.
    
- **Создание**: Когда ветка `develop` достигает состояния, готового к релизу, из нее создается ветка `release/*`. Имя ветки обычно соответствует номеру версии, например, `release/v1.2.0`.
    
- **Кто создает**: Как правило, тимлид или релиз-инженер.
    
- **Жизненный цикл**: После того как релизная ветка полностью протестирована и готова к выпуску, она вливается как в `master` (для фиксации новой продуктивной версии), так и обратно в `develop` (чтобы все исправления, сделанные в процессе подготовки релиза, попали в основную ветку разработки). После слияния ветка `release/*` удаляется.
    

### `hotfix/*` (ветки для срочных исправлений)

- **Назначение**: Быстрое исправление критических ошибок, обнаруженных в продуктивной версии. Эти ветки позволяют оперативно внести необходимые изменения, не мешая текущему процессу разработки в `develop` и подготовке следующего релиза.
    
- **Создание**: Ветки `hotfix/*` — единственные, которые создаются непосредственно из ветки `master`. Это гарантирует, что исправление базируется на точном коде, который находится в продакшене.
    
- **Кто создает**: Разработчики, ответственные за исправление критических ошибок.
    
- **Жизненный цикл**: После завершения исправления ветка `hotfix/*` должна быть влита как в `master` (для немедленного исправления на продакшене), так и в `develop` (чтобы гарантировать, что исправление будет включено в последующие релизы). Ветка `master` после слияния помечается новым тегом версии (например, `v1.2.1`).
    

### `refactor/*` (ветки для рефакторинга)

- **Назначение**: Улучшение существующего кода без изменения его внешнего поведения. Хотя классический Git Flow не выделяет `refactor` как отдельный тип ветки, его использование помогает семантически отделить задачи по улучшению кодовой базы от разработки новой функциональности.
    
- **Создание**: Чаще всего рефакторинг рассматривается как часть разработки и, следовательно, ветки `refactor/*` создаются из `develop`, подобно веткам `feature/*`. Например, `refactor/optimize-database-queries`.
    
- **Кто создает**: Разработчики.
    
- **Жизненный цикл**: Аналогичен `feature/*`. После завершения рефакторинга и прохождения код-ревью ветка вливается в `develop`.
    

### `bugfix/*` (ветки для плановых исправлений)

- **Назначение**: Исправление некритичных ошибок, которые могут подождать до следующего планового релиза. Важно отличать `bugfix` от `hotfix`: `hotfix` предназначен для срочных исправлений в продакшене, в то время как `bugfix` — для плановой работы над ошибками.
    
- **Создание**: Ветки `bugfix/*` создаются из `develop`. По сути, исправление некритичной ошибки — это такая же задача, как и разработка новой функциональности, и должна следовать тому же процессу.
    
- **Кто создает**: Разработчики.
    
- **Жизненный цикл**: Аналогичен веткам `feature/*`. После исправления бага и тестирования ветка `bugfix/*` вливается в `develop`.
    

## Правила создания коммитов

Чтобы история изменений была читаемой, понятной и информативной, в Yalla принят стандарт оформления коммитов, основанный на спецификации **Conventional Commits**. Такой подход не только улучшает навигацию по истории, но и позволяет автоматизировать создание списков изменений (changelogs) и управление версиями.

Каждое сообщение коммита должно иметь следующую структуру:

**`type(scope): subject`**

**type — Тип коммита (обязательно)**  
Тип должен соответствовать одному из следующих значений:

- **`feat`**: A new feature (`feature/*`).
    
- **`fix`**: A bug fix (`bugfix/*`, `hotfix/*`).
    
- **`refactor`**: A code change that neither fixes a bug nor adds a feature (`refactor/*`).
    
- **`docs`**: Documentation only changes.
    
- **`style`**: Changes that do not affect the meaning of the code (white-space, formatting, missing semi-colons, etc).
    
- **`test`**: Adding missing tests or correcting existing tests.
    
- **`chore`**: Other changes that don't modify source or test files (e.g., updating build tasks, package manager configs).
    

**scope — Область изменений (необязательно)**  
Указывает на часть кодовой базы, которую затронули изменения (например, `auth`, `api`, `ui-kit`).

**subject — Краткое описание (обязательно)**  
Это краткое, но емкое описание сути изменений.

- Пишется на **английском языке**.
    
- Используется повелительное наклонение (imperative mood): "Add", "Fix", "Update" вместо "Added", "Fixes".
    
- Начинается с заглавной буквы.
    
- В конце точка не ставится.
    
- Длина не должна превышать 50–72 символов.
    

**Примеры хороших коммитов:**

```text
feat(auth): Add authentication via Google OAuth
fix(api): Correct invalid date handling in /reports endpoint
refactor: Simplify logic in the pricing calculation service
docs: Update README with installation instructions
chore(deps): Bump react library to version 18.2.0
```

При необходимости можно добавить более подробное описание в теле коммита, отделив его от заголовка пустой строкой.

## Заключение

Представленный Git Flow и правила оформления коммитов обеспечивают четкую и структурированную модель управления кодом. Это помогает командам в Yalla эффективно сотрудничать и выпускать качественный продукт. Разделение работы на изолированные ветки, строгие правила слияния и ясное предназначение каждой ветки минимизируют риски и делают процесс разработки прозрачным и предсказуемым.
